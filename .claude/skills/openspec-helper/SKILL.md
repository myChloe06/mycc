# OpenSpec Helper

> Spec-Driven Development 助手 - "用嘴实现功能"的完全自动化版本
>
> 核心用法：你用自然语言描述需求 → 我自动调用所有命令 → 完成开发

---

## 核心原则

**你只需要**：用自然语言描述需求、做选择、确认继续

**我来做**：调用所有 OpenSpec 命令、生成 artifacts、实施代码

---

## 完整自动化流程

### 阶段 1：触发检测

当用户说以下内容时，主动提醒是否使用 OpenSpec：

**触发词**：
- "我想开发/做个/写个..."（新功能、新项目）
- "帮我做个/写个..."（新功能、新项目）
- "新增/添加..."功能
- "我要做一个新网站/新应用"
- "不知道从哪开始"

**我的回应**：
```
检测到你要开发新功能/新项目，要不要用 OpenSpec 来规范开发？

OpenSpec 的好处：
✅ 自动生成提案文档 (spec.md)
✅ 自动生成任务清单 (tasks.md) - 带勾选框
✅ 多工具协作 (Claude 写代码、Codex 审查)
✅ 一眼看进度 (看 tasks.md 打没打勾)

回复「用」开始，或说「不用」直接开发。
```

---

### 阶段 2：自动创建变更

**用户确认"用"后**：

**步骤 2.1：自动命名**
```
好的！我给这个功能起名：XXX（kebab-case）

确认这个名称可以吗？回复「可以」或「改成 XXX」
```

**步骤 2.2：用户确认后，自动调用 openspec-new-change**
- 不让用户输入任何命令
- 自动调用 Skill 工具执行 `openspec-new-change`
- 读取生成的文件状态

**步骤 2.3：显示状态**
```
✓ 变更已创建：openspec/changes/<name>/

下一步：确认需求细节
```

---

### 阶段 3：需求收集（提供选项）

**我问用户**：提供常见功能选项，不让用户盲目猜测

**示例（登录功能）**：
```
现在确认一下需求：

📋 常见功能选项：
☑ 邮箱密码登录
☑ 手机验证码登录
☑ 第三方登录（微信/GitHub）
☑ 记住我功能
☑ 找回密码
☑ 图形验证码
☑ 友好错误提示

☐ 你有其他特殊要求吗？（请补充）

回复：
- 「都要」- 以上所有功能都做
- 「除了 XXX」- 排除某些功能
- 或列出你想要的功能
```

**用户回复后**，我确认需求清单。

---

### 阶段 4：自动生成 Artifacts

**需求确认后**，我自动：

1. **调用 openspec-continue-change** 生成 spec.md
2. **调用 openspec-continue-change** 生成 tasks.md
3. **调用 openspec-continue-change** 生成其他 artifacts（如果有）
4. **显示任务清单**

**输出示例**：
```
✓ spec.md 已生成
✓ tasks.md 已生成

📋 任务清单（共 7 项）：
- [ ] 创建登录表单 UI
- [ ] 实现邮箱验证
- [ ] 实现密码验证
- [ ] 添加图形验证码
- [ ] 添加"记住我"功能
- [ ] 添加错误提示
- [ ] 编写单元测试

是否开始实施？回复「开始」或「继续」
```

---

### 阶段 5：自动实施

**用户确认"开始"后**：

1. **自动调用 openspec-apply-change**
2. **按 tasks.md 逐步实施**
3. **每完成一个任务，更新打勾状态**
4. **遇到问题停下来询问用户**

**输出示例**：
```
开始实施任务 1/7：创建登录表单 UI...

✓ 任务 1 完成
- [✓] 创建登录表单 UI
- [ ] 实现邮箱验证
...

继续任务 2/7：实现邮箱验证...
```

---

### 阶段 6：验证和归档

**所有任务完成后**：
```
所有任务已完成！🎉

是否验证实施？回复「验证」或直接「归档」
```

**用户说"验证"**：
- 自动调用 `openspec-verify-change`
- 显示验证结果

**用户说"归档"**：
- 自动调用 `openspec-archive-change`
- 完成流程

## 用户只用的自然语言

| 用户说 | 我做 |
|--------|------|
| "我想做个登录功能" | 检测触发，提醒用 OpenSpec |
| "用" | 自动命名、创建变更 |
| "可以" | 确认名称 |
| "都要" | 确认需求选项 |
| "开始" | 自动调用实施命令 |
| "继续" | 继续下一个任务 |
| "完成" | 验证并归档 |

---

## 关键规则

### ✅ 必须做的

1. **自动调用命令**：用 Skill 工具调用 openspec-* 命令，不让用户输入
2. **提供选项**：需求阶段给出常见功能选项
3. **自动命名**：根据需求自动生成 kebab-case 名称
4. **进度透明**：显示当前任务和剩余任务
5. **遇到问题问用户**：不确定时停下来确认

### ❌ 不能做的

1. **不让用户输入命令**：所有 `/opsx:xxx` 命令由我自动调用
2. **不让用户盲目猜**：需求时提供选项，不问开放式问题
3. **不让用户起名**：自动命名后确认即可
4. **不跳过确认**：每个阶段都要用户确认

## 注意事项

1. **不要强制使用**：如果用户说"不用"，就按正常方式开发
2. **新手友好**：第一次用可以建议先走 `/opsx:onboard` 教程
3. **用嘴实现**：目标是让用户只动嘴，不动手
4. **命令隐藏**：用户不需要知道任何 `/opsx:xxx` 命令
